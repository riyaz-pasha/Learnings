## Interview Guide: How to Recognize When Normal Sliding Window Doesn't Work

Most interview mistakes happen because people try to force the sliding window pattern on every subarray problem. Instead, ask these questions.

---

# Step 1: Is the condition monotonic?

A condition is **monotonic** if moving the left pointer can always restore validity.

### ✅ Yes → Classic Sliding Window

Examples:

* Sum ≤ K (positive numbers)
* At most K distinct elements
* No duplicate characters
* At most K odd numbers

Example:

```text
[1,2,3,4]   ❌ Invalid

Move left

[2,3,4]     ✅ Valid
```

**Rule:** If removing elements can only improve the condition, use a normal sliding window.

---

### ❌ No → Look for another pattern

Example (this problem):

```text
[1,3,5]     ✅ Valid

Move left

[3,5]       ❌ Invalid
```

Removing an element **destroyed** the validity.

A classic sliding window cannot decide when to shrink.

---

# Step 2: What determines validity?

This is usually the biggest clue.

### A. Counts/Frequencies → Sliding Window

The window state depends on **how many** of something exists.

Examples:

* At most K distinct
* Character frequencies
* Sum ≤ K
* Number of odd/even elements

Maintain:

* HashMap
* Frequency array
* Running sum

---

### B. Positions/Required Elements → Last Seen Indices

The window depends on **where important elements occur**, not how many.

Examples:

* Must contain `minK`
* Must contain `maxK`
* Must contain all vowels
* Must contain A, B, and C
* Latest occurrence of X

Maintain only the **latest positions** of important elements.

---

# Step 3: What is the question asking?

### Longest / Shortest

Usually:

> Maintain one valid window.

Think:

* Sliding Window
* Two Pointers

---

### Count All Subarrays

Instead of finding every subarray, ask:

> **How many valid subarrays end at index `i`?**

This often leads to:

* Last Seen Indices
* Prefix Sum
* Contribution counting

---

# Step 4: What is the minimum information needed?

Don't automatically store the whole window.

Ask:

> **Can I answer the question using just a few variables?**

For this problem:

```text
lastBad  = last index with an invalid value
lastMin  = last occurrence of minK
lastMax  = last occurrence of maxK
```

Nothing else matters.

Contribution at each index:

```text
validStarts = max(0, min(lastMin, lastMax) - lastBad)
```

---

# Pattern Recognition Cheat Sheet

| Problem Characteristics    | Typical Pattern       |
| -------------------------- | --------------------- |
| Longest/Shortest subarray  | Sliding Window        |
| At most K                  | Sliding Window        |
| Counts/Frequencies         | Sliding Window        |
| Exact sum                  | Prefix Sum + HashMap  |
| Divisible by K             | Prefix Sum            |
| Contains required elements | Last Seen Indices     |
| Count all valid subarrays  | Contribution Counting |
| Next Greater/Smaller       | Monotonic Stack       |

# Golden Rules

### ✅ Use Sliding Window when:

* The condition is **monotonic**.
* Removing elements can only improve validity.
* The window state depends on **counts, frequencies, or sums**.

---

### ✅ Use Last Seen Indices when:

* The condition depends on the **latest occurrence** of required elements.
* Removing elements may **break** an otherwise valid window.
* You only need a few important positions instead of the entire window.

---

### ✅ Use Contribution Counting when:

Instead of asking:

> "Which subarrays are valid?"

Ask:

> **"How many valid subarrays end at each index?"**

This shift in perspective often transforms an O(n²) solution into an O(n) one.

---

# The One Rule to Remember

> **If validity can always be restored by moving the left pointer, use a classic Sliding Window. If moving the left pointer can both fix *and* break validity, a classic Sliding Window is usually the wrong choice. Look for Last Seen Indices, Prefix Sums, or another pattern instead.**

This mental model is one of the most reliable ways to identify the correct approach during coding interviews.

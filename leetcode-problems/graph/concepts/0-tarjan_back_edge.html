Before processing back edge 5 → 1, look at vertex 1's values:

Section 03

disc[v] vs low[v] — Side by Side

Back edge: vertex 5 → vertex 1 (vertex 1 is already visited, not parent)

✓ CORRECT — use disc[1]

low[u] = min(low[u], disc[v])

low[5] = min(5, disc[1]) = min(5, 1) = 1

What this says:

Vertex 5 can reach back to the vertex discovered at time 1, which is vertex 1. Nothing more. The back edge is a direct jump to vertex 1.

✓ Accurately captures only what the back edge provides

✗ WRONG — using low[1]

low[u] = min(low[u], low[v])

low[5] = min(5, low[1]) = min(5, 0) = 0

What this wrongly claims:

Vertex 5 can reach back to the vertex discovered at time 0 (vertex 0) just by taking this back edge. But vertex 5 can only directly jump to vertex 1 via this edge.

✗ "Steals" reachability from vertex 1's other subtree

Section 04

Why low[v] is Wrong — The Core Reasoning

When vertex 5 takes the back edge to vertex 1, the back edge contributes one thing only: a direct connection to vertex 1. That's it.

Vertex 1's low[1] = 0 was earned because vertex 1 has a whole separate subtree (vertices 2 and 3) where vertex 3 has a back edge all the way back to vertex 0. That is vertex 1's reachability through its left subtree — not through vertex 5.

Reachability Paths — How vertex 5 reaches vertex 0
✓ Correct Path
5 →(back edge) 1 →(tree edge) 2 →(tree edge) 3 →(back edge) 0
Notice: vertex 5 reaches vertex 0 by:
1. Taking the back edge to vertex 1 (contributes disc[1] = 1)
2. Then using tree edges downward through vertex 2, vertex 3
3. Then using vertex 3's back edge up to vertex 0

Step 1 is what the back edge contributes. Steps 2–3 are propagated upward through low[child] tree-edge propagation. Using low[1] = 0 for step 1 skips steps 2–3 and "steals" credit from vertex 3's back edge.
💡 The Invariant That low[] Maintains

low[u] = minimum discovery time reachable from u's subtree using DFS tree edges + direct back edges within the subtree.

A back edge from u → v directly contributes disc[v] to low[u]. The additional reachability from v's other subtrees is propagated upward through tree edges when the DFS returns — that's what low[u] = min(low[u], low[child]) handles.

If we use low[v] for back edges, we're conflating two separate mechanisms into one. The back edge propagation and the tree-edge propagation get tangled together.

Section 05

Does Using low[v] Give Wrong Bridges?

This is the honest answer: for simple undirected graphs with correct parent tracking, using low[v] happens to produce correct bridge detection in the end — because low[v] ≤ disc[v] always, so using low[v] can only ever lower low values, and a mathematical argument shows this doesn't create false non-bridges.

But the intermediate values are wrong, and here's why that matters:

Why disc[v] is still correct
1
The low[] invariant stays semantically pure — it means exactly what it says: "earliest vertex directly reachable from this subtree"
2
For directed graphs (Tarjan's SCC algorithm), using low[v] IS provably wrong because you can't walk back up through v's subtree
3
For multigraphs (parallel edges), the two can diverge and give incorrect bridge counts
4
Conceptual correctness matters: reasoning about the algorithm's correctness becomes much harder when invariants are violated mid-computation
Final values in our example

Even with low[v], bridges are correctly detected here. But look at low[5]:

using disc[1]
low[5] = 1
"5 reaches vertex 1"
using low[1]
low[5] = 0
"5 reaches vertex 0"

low[5] = 0 is technically not wrong (vertex 5 CAN reach vertex 0), but it arrived there by "borrowing" vertex 1's low value instead of earning it properly through tree-edge propagation. The invariant is broken.

Section 06

The One-Line Summary

A back edge u → v means vertex u can jump directly to v. So it contributes disc[v] to low[u] — the discovery time of v.

low[v] is what v can reach through v's own subtree's back edges — that information is already woven into v's subtree's low[] values, and it will propagate upward through the tree naturally via low[u] = min(low[u], low[child]).

Using low[v] double-counts: it applies v's "already-earned" reachability to u's position as if u's single back edge bought all of it.

Section 07

Back in the Code

// CASE 1: Tree edge → propagate low from child's subtree
low[u] = Math.min(low[u], low[v]); // ← use low[v]: child found a path via its subtree

// CASE 2: Back edge → direct jump to an ancestor
low[u] = Math.min(low[u], disc[v]); // ← use disc[v]: only what THIS edge directly buys us
⚠ The Temptation

Since low[v] also gives correct bridges in simple undirected graphs, many online implementations use it. But on exams, interviews, and directed graph extensions, the distinction matters. The code in your question is correct — disc[v] is the right choice.
